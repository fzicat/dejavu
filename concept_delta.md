# DejaVu v2 — Product/UX Delta (Locked Decisions)

## 1. Data Layer Decision
**Decision:** Free data sources only. Primary: Alpaca IEX (intraday). Fallback: `yahooquery` if Alpala IEX is unavailable or rate-limited.
**Implication:** All data-fetching commands (`use [TICKER]`) must display a source indicator (e.g., `[Alpaca IEX]` or `[yahooquery]`) in the status line. Cache metadata includes the source tag so operators know data fidelity. `yahooquery` fallback may provide coarser timestamps or delayed bars — the CLI will emit a subtle warning (e.g., `⚠ yahooquery fallback — bars may be delayed`).

## 2. CLI Framework Decision
**Decision:** `prompt_toolkit` as the interactive shell engine.
**Implication:** The REPL will feature:
- Tab-completable subcommands (e.g., typing `model` + TAB → completes to `model hmm`, `model ar`, `model rnn`).
- History persistence across sessions (`.dejavu_history`).
- Syntax highlighting for commands and inline parameter tooltips.
- Multi-line strategy rule entry via continuation prompts (`>>>` style).

## 3. Slippage/Commission
**Decision:** Zero slippage, zero commission for MVP.
**Implication:** All backtest metrics reflect raw signal performance, not realistic execution. A disclaimer appears in every `metrics` card: `⚠ MVP: assumes zero slippage/commission.` This keeps the MVP scope tight but operators must interpret results as alpha potential, not live expectancy.

## 4. Multi-Rule Strategy Support (MVP)
**Decision:** Multi-rule strategies are fully supported in MVP.
**UX Implications:**
- Strategy creation is now a mini-workflow within the REPL:
  ```
  strategy new my_strategy
  rule 1: IF state=up_trend AND price > vwap THEN long
  rule 2: IF state=choppy AND price > prev_close THEN long
  rule 3: EXIT at session_close
  strategy save
  ```
- The `strategy list` command shows rule counts per strategy.
- The `strategy show [name]` command prints the full rule cascade, including any exit conditions.
- Multi-rule conflict resolution: if two rules trigger simultaneously on the same bar, the first-defined rule takes precedence. This is documented but not resolved programmatically in MVP.
- Validation: `strategy save` runs a dry-validation pass and reports any rules that never trigger on the cached date range (dead rules).

## 5. Auto-Labeled States (MVP)
**Decision:** States are automatically labeled with descriptive names based on their statistical properties.
**UX Implications:**
- After `model hmm --states 3` completes, the CLI prints:
  ```
  States inferred:
  0: "Up-Trend"      (mean_return=+0.12%, vol=0.08%, duration=12 bars)
  1: "Choppy"        (mean_return=-0.01%, vol=0.15%, duration=5 bars)
  2: "Down-Trend"    (mean_return=-0.10%, vol=0.11%, duration=8 bars)
  ```
- Labels are generated by a simple heuristic engine (not LLM-based) examining: mean return sign, volatility relative to average, and average state duration.
- Labels appear in `show states`, `show summary`, and throughout the strategy builder: `rule 1: IF state="Up-Trend" AND ...`
- Operators can override auto-labels with `model relabel --state 0 "MyCustomName"` for personalization.
